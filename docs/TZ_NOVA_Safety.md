## 1. Назначение и цели

### 1.1. Назначение

Веб-приложение **NOVA Safety AI** автоматизирует полный цикл оформления и исполнения **наряд-допуска на производство работ (НДПР)** в соответствии с процедурами UOG (в т.ч. UOG-HSE-PR-007, UOG-HSE-PR-001 и связанные бланки):

1. Загрузка и разбор **ППР** (в т.ч. с помощью ИИ).
2. Оформление **НДПР** (реквизиты, бригада, сроки, фото участка).
3. Формирование **АСОР / АБР / оценки рисков**.
4. Спецразрешения на огневые / газоопасные / работы в замкнутом пространстве.
5. Согласование с **ЭЦП eGov** (SIGEX / QR / NCALayer).
6. Выдача, выполнение, остановка и закрытие наряда.
7. Журнал, уведомления, администрирование.

### 1.2. Цели внедрения

| № | Цель | Критерий успеха |
|---|------|-----------------|
| 1 | Единый цифровой пакет вместо бумаги | ППР → НДПР → риски → разрешения → ЭЦП → PDF |
| 2 | Юридически значимая подпись | ЭЦП eGov по ролям, хэш документа, проверка CMS |
| 3 | Контроль очереди согласования | Производитель → [ERT] → допускающий → выдающий → утверждающий |
| 4 | Работа с телефона на объекте | PWA, адаптивный UI, камера |
| 5 | Аудит | Статусы, отклонения, журнал, уведомления |

### 1.3. Нормативная и методическая база

- Внутренние процедуры HSE UOG (PR-007, ASOR F01 и др.).
- Электронная подпись РК (NCALayer / eGov Mobile / SIGEX).
- Оценка рисков в логике NEBOSH / матрицы в составе АСОР.

## 3. Роли пользователей

| Роль (код) | Назначение |
|------------|------------|
| `coordinator` | Админ HSE: пользователи, журнал, удаление (кроме черновиков), перенумерация, полный доступ |
| `performer` | Производитель работ: создание пакета, отправка, ЭЦП, работы, закрытие |
| `permitter` | Допускающий: раздел 3 разрешений, ЭЦП, остановка/закрытие |
| `issuer` | Выдающий НД: ЭЦП, выдача, остановка/закрытие |
| `leadExpert` | Утверждающий (кат. 1 / огневые): ЭЦП |
| `ert` | ПАС / ERT: газотест, ЭЦП при огневых/газоопасных |
| `executor` | Работник бригады: ознакомление с АБР/оценкой рисков (ЭЦП) |
| `safety` | Инспектор: остановка работ, аннулирование, разбор отклонений |
| `contractor` | Подрядчик: оформление пакета подрядных нарядов |

Вход: email + пароль (Firebase Auth). Профиль: документ Firestore `users/{uid}` (`displayName`, `role`, `email`, опционально `notificationEmail`).

## 4. Функциональные требования (реализовано)

### 4.1. Журнал нарядов

- Список нарядов с фильтрами (в работе / архив / поиск).
- Задания текущего пользователя: подписи, газотест ERT, проверки допускающего, ознакомление бригады.
- Создание пакета: «с ППР» / мастер шагов.
- Карточка наряда: статус, PDF-пакет, очередь ЭЦП, отклонение, остановка работ.
- **Черновики нельзя удалять** из UI (защита от потери номера при пересогласовании).

### 4.2. Шаг «ППР»

- Загрузка DOC / DOCX / PDF.
- Извлечение данных ИИ (описание, этапы, меры контроля) через Cloud Functions.
- Ручное редактирование и переход к НДПР.
- Сохранение вложения и данных в сессии / документе наряда.

### 4.3. Шаг «НДПР»

- Реквизиты объекта, виды работ, зона, описание, этапы, объём, инструменты.
- Назначение участников: производитель, допускающий, выдающий, утверждающий, ERT.
- Бригада (исполнители), сроки, чек-лист НДПР (кат. 1).
- Регистрационный номер (`001`, `002`, …); при отклонении и повторной подаче **номер сохраняется**.
- Фотофиксация участка (до 6 фото, сжатие, подпись к фото).
- Автосохранение черновика в `sessionStorage`.

### 4.4. Шаг «Оценка рисков / АСОР / АБР»

- АСОР (мероприятия ОТ, ТБ, ООС): ручной ввод и/или генерация ИИ.
- АБР: формирование, ручная проверка, PDF.
- Оценка рисков (матрица / задания / опасности) — PDF в составе пакета.
- Ежедневное ознакомление с АБР (смена).

### 4.5. Шаг «Разрешения» (при необходимости)

Спецразрешения по видам работ:

| Вид | Код | Примечание |
|-----|-----|-----------|
| Огневые работы | `open_flame_fire` | Газотест, согласование ERT |
| Газоопасные | `gas_hazard` | Газотест, согласование ERT |
| Замкнутое пространство | `confined_space` | Чек-листы типов/спасения |

- Производитель заполняет «Требуется».
- Допускающий на карточке наряда заполняет раздел 3 («Имеется») — **опционально для открытия ЭЦП**, данные входят в PDF.
- PDF разрешений и закрытие участка после завершения работ.

### 4.6. Отправка и согласование

1. «Отправить и подписать»: сохранение пакета → ЭЦП производителя (в т.ч. на черновике) → статус `on_approval`.
2. Создание приглашений подписантам (`provisionPermitSigners`).
3. Ознакомление бригады с АБР и оценкой рисков (ЭЦП всех исполнителей) — блокирует следующих согласующих.
4. Очередь ЭЦП:
   - обычные работы: производитель → допускающий → выдающий → утверждающий;
   - огневые/газоопасные: производитель → **ERT** → допускающий → выдающий → утверждающий.
5. ERT вводит результаты газотеста до своей подписи.
6. Автовыдача при завершении обязательных подписей (и/или ручная выдача).
7. Отклонение с комментарием; производитель исправляет и отправляет снова (**тот же № НДПР**).

### 4.7. Исполнение и закрытие

Статусы: `draft` → `on_approval` → `issued` → `in_progress` ↔ `suspended` → `closed` → `archived`; ветки `rejected`, `annulled`.

- Старт работ, приостановка, закрытие НДПР.
- Остановка работ (work stop): причина, фото, уведомление инспектора; снятие / аннулирование.
- Раздел закрытия разрешений в PDF.

### 4.8. Электронная подпись

- eGov Mobile / Business (QR), NCALayer.
- Подпись единого PDF-пакета; серверная запись CMS / метаданных.
- Опциональная проверка ФИО по сертификату (настройка админа).
- Отдельные подписи на бланках спецразрешений (производитель / выдающий / допускающий).

### 4.9. Уведомления

- Push (FCM) при приглашениях на подпись, событиях наряда, остановке работ.
- Email (SMTP) на `notificationEmail` или email учётки.
- Внутриприложенные приглашения (`signingInvites`) и уведомления (`permitNotices`).

### 4.10. Администрирование

- CRUD пользователей (роли, email уведомлений) через Cloud Functions.
- Журнал нарядов: просмотр, выборочное удаление (не черновиков), очистка «осиротевших» приглашений.
- Перенумерация нарядов (админ).
- Экспорт журнала (CSV).
- Настройки подписания / инспектора.

### 4.11. ИИ-ассистент

- Плавающий чат «ИИ» на экранах приложения.
- Вызовы только через Cloud Functions (ключ модели не в клиенте).

## 10. Ограничения и допущения

- Саморегистрация пользователей на проде отключена; учётки создаёт администратор.
- Полная доставляемость email требует корректного SMTP и (для борьбы со спамом) домена с SPF/DKIM — выносится в эксплуатационную настройку.
- Кастомный домен Hosting — опционально, вне обязательного объёма стенда `*.web.app`.
- Юридическая сила ЭЦП определяется применимым законодательством РК и политикой Заказчика; система обеспечивает технический контур подписи и хранения факта подписи.
